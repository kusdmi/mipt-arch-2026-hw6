# Currency Converter Refactoring

В исходном проекте был реализован конвертер валют из USD в несколько валют (EUR, GBP, RUB, CNY).  
Хотя программа выполняла свою задачу, её архитектура содержала ряд проблем, связанных с поддерживаемостью, масштабируемостью и качеством кода.

В рамках рефакторинга были выявлены и исправлены нарушения следующих принципов разработки:

- SOLID
- DRY (Don't Repeat Yourself)
- KISS (Keep It Simple, Stupid)
- YAGNI (You Aren't Gonna Need It)
- принципов Clean Code

Также была упрощена архитектура приложения и устранено значительное количество дублирующегося кода.

---

# Найденные проблемы и их решения

## 1. Нарушение Interface Segregation Principle (SOLID)

### Проблема

Интерфейс `CurrencyConverter` требовал реализации методов для всех типов конвертации:

```
convert_usd_to_eur
convert_usd_to_gbp
convert_usd_to_rub
convert_usd_to_cny
```

Каждый конкретный конвертер был вынужден реализовывать методы, которые фактически не использовались.

Пример:

```
def convert_usd_to_gbp(self, amount):
    print("This is not USD to GBP converter")
```

Это приводило к следующим проблемам:

- появлению "пустых" методов
- ухудшению читаемости кода
- нарушению принципа разделения интерфейсов (ISP)

### Решение

Интерфейс был переработан и теперь отвечает только за получение курсов валют:

```
class RateProvider(ABC):
    def get_rates(self) -> dict
```

Конвертация выполняется отдельным универсальным классом `CurrencyConverter`.

## 2. Нарушение DRY

### Проблема

Логика получения курсов валют повторялась в нескольких файлах:

- usd_eur_converter.py
- usd_gbp_converter.py
- usd_rub_converter.py
- usd_cny_converter.py

Каждый из этих файлов содержал одинаковые блоки кода:

- HTTP запрос к API
- парсинг JSON
- обработку ошибок
- retry логику
- настройку логгера

Это усложняло поддержку проекта, так как изменение API требовало редактирования нескольких файлов.

### Решение

Создан отдельный класс:

```
ApiRateProvider
```

который централизованно выполняет:

- получение курсов валют
- обработку ошибок
- retry механизм
- логирование

Теперь изменение API требует модификации только одного файла.

---

## 3. Нарушение Single Responsibility Principle (SOLID)

### Проблема

Классы конвертеров выполняли несколько разных задач одновременно:

- получали данные из API
- выполняли конвертацию
- логировали ошибки
- иногда работали с кешем

Это нарушает принцип единственной ответственности (SRP).

### Решение

Функциональность была разделена между двумя классами:

```
ApiRateProvider
    отвечает за получение курсов валют

CurrencyConverter
    выполняет только конвертацию
```

Теперь каждый класс имеет одну чёткую ответственность.

---

## 4. Нарушение Open/Closed Principle (SOLID)

### Проблема

Чтобы добавить новую валюту, например JPY, требовалось:

1. Создать новый файл
2. Создать новый класс
3. Реализовать несколько методов конвертации

Это усложняло расширение системы.

### Решение

Вместо отдельных методов для каждой валюты реализован универсальный метод:

```
convert(amount, currency)
```

Теперь добавление новой валюты не требует изменения кода.

---

## 5. Нарушение KISS

### Проблема

Для каждой валюты был создан отдельный класс:

- UsdEurConverter
- UsdGbpConverter
- UsdRubConverter
- UsdCnyConverter

Это усложняло архитектуру системы и увеличивало количество файлов.

### Решение

Создан один универсальный класс:

```
CurrencyConverter
```

который может работать с любой валютой.

---

## 6. Нарушение YAGNI

### Проблема

В каждом классе присутствовали методы, которые фактически никогда не использовались:

```
convert_usd_to_eur
convert_usd_to_gbp
convert_usd_to_rub
convert_usd_to_cny
```

Большинство этих методов просто выводили сообщение о том, что они не предназначены для данной валюты.

### Решение

Оставлен один универсальный метод:

```
convert(amount, currency)
```

Это значительно уменьшило количество кода.

---

## 7. Проблемы Clean Code

### Обнаруженные проблемы

В исходном коде использовались:

- `print` вместо обработки ошибок
- дублирование логики
- неочевидные зависимости между классами
- отсутствие проверки данных
- смешивание ответственности классов

### Исправления

В новой версии:

- используются исключения (`RuntimeError`, `ValueError`)
- реализован централизованный логгер
- добавлены проверки корректности данных
- логика получения курсов вынесена в отдельный компонент
- улучшена читаемость и структура кода

# Итоговая архитектура проекта

После рефакторинга структура проекта выглядит следующим образом:

```
project/

currency_converter.py
    RateProvider (абстрактный интерфейс)
    CurrencyConverter (основная логика конвертации)

api_rate_provider.py
    ApiRateProvider (получение курсов валют из API)

main.py
    точка входа в программу
```

Архитектура разделяет ответственность между компонентами и упрощает расширение системы.

# Пример использования

```
from api_rate_provider import ApiRateProvider
from currency_converter import CurrencyConverter

provider = ApiRateProvider()
converter = CurrencyConverter(provider)

result = converter.convert(10, "RUB")

print(result)
```

Пример конвертации:

```
10 USD -> 913.4 RUB
```

(значение зависит от актуального курса API)
